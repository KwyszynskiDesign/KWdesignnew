Kontekst techniczny

Aplikacja zbudowana jako TypeScript monorepo bez frameworka frontendowego. Decyzja wynikała z wymagań środowiskowych: drukarnia nie ma serwera, nie ma IT, aplikacja musi działać jako lokalny plik HTML otwierany z dysku lub z GitHub Pages. Zero Node.js w środowisku produkcyjnym — tylko przeglądarka.

Kluczowe wyzwanie architektoniczne: 16 kategorii cennikowych z różnymi logikami (progi m², progi szt., formaty flat, cenniki mb × długość, dopłaty procentowe) musi być zdefiniowanych w danych, nie w kodzie — żeby pracownicy mogli zmieniać ceny bez redeploymentu.

Proces techniczny

Problem wyjściowy: CSV z cennikiem zawierał 16 sekcji z zupełnie różną strukturą — raz tabela progów m², raz lista formatów, raz dopłaty procentowe z komentarzami tekstowymi w komórkach.

  • Zaprojektowanie TypeScript interface PriceTable — unit, pricing mode (per-m², per-szt., per-format, per-mb), tiers[], modifiers[], rules, notes
  • Ręczna normalizacja 16 sekcji CSV → pliki data/normalized/*.json
  • Walidacja schematów przez zod przy każdym starcie aplikacji — błąd w JSON = błąd widoczny w konsoli, nie ciche złe obliczenie
  • Zasada: nowa kategoria = nowy plik JSON — zero modyfikacji istniejących struktur

Problem: 16 modułów TypeScript z hardcoded cenami — każda zmiana wymagała redeploymentu, a pracownicy nie mają dostępu do kodu.

  • Singleton z lazy loading: moduł ładuje JSON jednorazowo i cache'uje w pamięci — kolejne wywołania bez fetch
  • resolveStoredPrice(key, defaultValue): sprawdza localStorage[key] przed użyciem wartości z JSON — override bez dotykania kodu
  • Namespace klucze: razdwa_prices.artykuly-plakaty.a0-czb — hierarchiczne, bez kolizji
  • StorageEvent: zmiana localStorage w jednej karcie → event broadcast → inne karty przeliczają ceny bez reloadu
  • Mockowanie: vi.mock('./priceService') — każdy test ma kontrolowane ceny, zero interferencji między testami

Wymaganie: PDF z zamówieniem bez serwera — drukarnia nie ma backendu, a pracownicy muszą wysyłać PDF klientom mailowo.

  • pdf-lib 1.17 + fontkit: tworzenie dokumentu PDF w przeglądarce z polskimi znakami (font embedded)
  • Zawartość PDF: nagłówek z logo drukarni, tabela pozycji z cenami, linia rabatu, linia EXPRESS, suma końcowa, dane klienta, data/numer zamówienia
  • Blob download: URL.createObjectURL(blob) → a.click() — pobieranie bez otwierania nowej karty
  • Offline: generowanie PDF działa bez sieci — font i cały pdf-lib w bundlu (523 kb)

Wymaganie: historia wszystkich zamówień w Google Sheets — pracownicy znają arkusze, nie chcą nowej bazy danych.

  • GAS Web App: doPost(e) przyjmuje JSON, parsuje i appenduje wiersz do arkusza
  • fetch() no-cors: klient wysyła zamówienie POST bez obsługi odpowiedzi (ograniczenie no-cors mode) — fire and forget
  • Payload: timestamp, lista pozycji (nazwa, ilość, cena jednostkowa, suma), rabat %, tryb, klient, suma końcowa
  • Retry logic: toast informuje o statusie — przy błędzie sieci użytkownik może wysłać ponownie

Problem: drukarnia ma niestabilne wifi — aplikacja musi działać bez sieci, ale po wdrożeniu nowej wersji musi pokazywać świeży build.

  • NetworkFirst dla app.js: każde otwarcie próbuje pobrać świeży bundle — dopiero przy braku sieci serwuje cache
  • CacheFirst dla statycznych assetów: CSS, fonty, ikony — stałe pliki cache'owane agresywnie
  • CACHE_NAME versioning: zmiana wersji w sw.js → activate event usuwa stary cache — zero przestarzałych zasobów
  • Manifest PWA: ikona, name, display:standalone — instalacja na pulpicie jednym kliknięciem z Chrome
